Fases de Execução
01
Fundação
Infraestrutura, Auth & Data Model
Nada funciona sem um banco bem modelado e autenticação sólida. Essa fase é o alicerce de tudo. Erros aqui custam refatoração cara depois.
2 semanas
Projeto & Repositório
Criar repositório GitHub + estrutura Next.js
App Router, TypeScript, ESLint. Configurar Vercel com deploy automático no push para main. Branch strategy: main (prod) + dev (staging).
Instalar e configurar Design System EOS
Tokens CSS, componentes base (Button, Card, Label, Badge, Toast), layout shell com nav fixo. Tudo do EOS Design System v1 implementado antes de qualquer tela.
Supabase — Data Model Completo
Auth + profiles
Supabase Auth nativo. Tabela profiles com user_id FK → auth.users. RLS ativado desde o dia 1 — cada usuário lê e escreve apenas seus próprios dados.
family_members
id, profile_id, name, age, medical_conditions (array), mobility, priority. Relação com profiles via FK. RLS herdado.
resources (inventário flexível)
Substituir campos fixos por modelo de itens: id, profile_id, canonical_key, name, category, quantity, unit, acquired_at. canonical_key é o elo de deduplicação cross-cenário.
scenarios
id, profile_id, type (enum tipado), duration_days, severity, special_conditions (array), status, created_at. Múltiplos cenários por profile. Um usuário pode ter Furacão Cat 4 + EMP + Pandemia simultâneos.
checklist_templates + checklist_items
templates: id, scenario_id, generated_at, profile_snapshot (JSON), is_stale. items: id, template_id, name, quantity, unit, tier (essential/moderate/excellent), category, rationale, source_reference, canonical_key, order.
circles + circle_members + circle_strength
circles: id, name, leader_id, invite_code, visibility, created_at. circle_members: role (leader/member/pending). circle_strength: score calculado via trigger, atualizado a cada mudança de inventário de qualquer membro.
⚠ RLS (Row Level Security) deve ser configurado em todas as tabelas nessa fase. Adicionar depois é tecnicamente possível mas arriscado e trabalhoso.
Entregáveis da Fase 1
Repositório configurado com CI/CD funcionando
Design System implementado como componentes reutilizáveis
Todas as tabelas criadas no Supabase com RLS ativo
Auth funcionando (signup, login, logout, sessão persistida)
Migrations versionadas no repositório
02
Produto
Profile, Família & Cenários
As telas de configuração. O usuário constrói quem ele é e o que pode acontecer. Essa é a coleta de contexto que alimenta toda a inteligência posterior.
2 semanas
Tela — Profile Setup
Formulário de perfil principal
Nome, localização (cidade/estado/tipo de região), tipo de moradia (casa/apartamento/rural/off-grid). Salva em profiles.
Gerenciamento de membros da família
Adicionar/editar/remover membros. Por membro: nome, idade, condições médicas (chips selecionáveis: diabetes, hipertensão, mobilidade reduzida, etc.), prioridade. Interface com cards, não tabela.
Tela — Cenários
Seleção de cenários tipados
Grade de cards com os tipos: Furacão, Terremoto, Enchente, Apagão, Racionamento de Água, EMP, Fallout, Pandemia, Incêndio Florestal, Crise Civil, Evacuação. Cada card tem ícone, nome e descrição curta. Usuário pode ativar múltiplos.
Configuração por cenário
Ao selecionar um cenário: slider de duração (1–90 dias), severidade (leve/moderado/severo), condições especiais detectadas automaticamente do profile (ex: "Membro diabético identificado — insulina adicionada à lista crítica").
Dependência: Fase 1 completa. Todas as operações CRUD desta fase gravam diretamente no Supabase via client-side com RLS.
Entregáveis da Fase 2
Usuário consegue criar e editar seu perfil completo
Usuário consegue adicionar membros da família com condições médicas
Usuário consegue ativar múltiplos cenários com parâmetros
Dados persistidos no Supabase e visíveis entre sessões
03
Inteligência
Motor de Decisão & Geração de Checklists
O coração do EOS. A LLM lê o perfil completo e gera checklists prescritas por cenário, em 3 tiers, com quantidades calculadas por pessoa e justificativas rastreáveis por fonte.
3 semanas
Rules Engine (determinístico)
Regras de prioridade e escalada
Implementar antes da LLM. Exemplos: IF membro_diabético → insulina entra no tier Essential obrigatoriamente. IF mobilidade_reduzida → plano de evacuação não pode incluir caminhada longa. IF crianças_menores_5 → água mínima sobe 20%. Essas regras nunca são sobrescritas pela LLM.
Prompt 1 — Geração de Checklist
Arquitetura do prompt e output estruturado
Input: profile completo (família, condições médicas, localização, moradia) + tipo de cenário + duração + severidade. Output: JSON com array de itens por tier. Cada item tem: name, quantity (calculada), unit, category, rationale, source_reference, canonical_key.
Cálculo de quantidade por pessoa
A LLM recebe a composição familiar e calcula: água = (adultos × 2L + crianças × 1.5L + idosos × 2L) × dias × fator de severidade. Mesma lógica para calorias, medicamentos, combustível. Os números aparecem com a conta explícita no campo rationale.
Serverless function — /api/generate-checklist
Recebe scenario_id + profile_id. Busca dados do Supabase. Monta contexto. Chama Claude API. Salva template + items no banco. Retorna template_id. Timeout: 30s. Fallback: servir regras determinísticas puras sem LLM.
Tela — Checklist por Cenário
Visualização em 3 tiers
Tabs ou acordeão: Essencial / Moderado / Excelente. Cada item mostra: nome, quantidade, unidade, categoria (chip colorido), e ao expandir: rationale + fonte. Barra de progresso por tier no topo.
Marcar item como adquirido
Tap no item → status muda para acquired → grava em user_inventory via canonical_key → progresso do tier recalcula em tempo real. Item adquirido em um cenário reflete automaticamente em todos os outros cenários que usam o mesmo canonical_key.
⚠ O fallback sem LLM deve ser implementado antes de ir para produção. Em uma emergência real, timeout da API não é aceitável.
Entregáveis da Fase 3
LLM gera checklist prescrita com quantidades calculadas por pessoa
Itens divididos em Essencial / Moderado / Excelente com justificativa
Marcar item como adquirido atualiza o inventário global
Cross-scenario deduplication funcionando via canonical_key
Fallback determinístico implementado e testado
04
Inteligência
Progressão Inteligente & Knowledge Base
A LLM passa de geradora de listas para conselheira contínua. A cada item adquirido, o sistema reavalia e sugere o próximo passo mais impactante — incluindo novos cenários.
3 semanas
Knowledge Base
Pipeline de indexação de documentos
Fontes: FEMA Emergency Supply List, CDC Emergency Preparedness, Red Cross Disaster Handbook, Military Survival FM 21-76. Pipeline: PDF → extração de texto → chunking por seção → embedding (text-embedding-3-small) → armazenamento em pgvector no Supabase. Chunks tagueados por scenario_type e item_category.
Retrieval integrado ao Prompt 1
Antes de chamar a LLM, buscar os chunks mais relevantes da knowledge base para o cenário + categorias presentes no profile. Injetar no contexto do prompt. Isso é o que torna as recomendações rastreáveis por fonte.
Prompt 2 — Avaliação de Progresso
Análise contínua do inventário
Triggered a cada item marcado. Input: checklist completa com status + inventário atual. Output: nível de preparação por tier (%), gaps críticos identificados, próximo item de maior impacto, quais cenários adicionais já estão parcialmente cobertos com o inventário atual.
Prompt 3 — Cross-Scenario Intelligence
Triggered ao completar um tier ou um cenário. Input: todos os cenários ativos + inventário agregado. Output: mapa de cobertura (quais itens servem múltiplos cenários), sequência ótima de aquisição, sugestão do próximo cenário baseado em overlap.
Dashboard Multi-Cenário
Visão geral de todos os cenários ativos
Card por cenário com: nome, barra de progresso por tier (Essencial X% / Moderado X% / Excelente X%), próxima ação recomendada, badge de status. Ordenado por criticidade calculada.
Entregáveis da Fase 4
Knowledge base indexada e integrada ao sistema de geração
Sistema avalia progresso e sugere próximo passo após cada aquisição
Cross-scenario intelligence identifica sequência ótima de compra
Dashboard mostra todos os cenários com progresso em tempo real
05
Social
Círculos — Resiliência Coletiva
A camada social do EOS. Famílias formam círculos privados, somam inventários, e constroem força coletiva medida em preparação real — não em número de membros.
3 semanas
Criação e Gestão de Círculos
Criar e configurar um Círculo
Nome, visibilidade (privado/descobrível), geração de invite_code único. O criador torna-se líder automaticamente. Interface de gestão de membros com roles.
Fluxo de entrada — solicitação e aprovação
Candidato usa invite_code ou descobre o círculo. Sistema calcula e exibe o preparedness score do candidato para o líder antes da decisão. Líder vê: score geral, cenários cobertos, e impacto que o candidato teria na força do círculo se aceito. Aceitar → membro. Rejeitar → notificação.
Circle Strength Score
Cálculo do score coletivo
Score = Σ (preparedness_score de cada membro × peso do cenário). Recalculado via Supabase trigger a cada mudança de inventário de qualquer membro. Armazenado em circle_strength com breakdown por tier e cenário.
Capability badges
Em vez de apenas número, exibir o que o círculo consegue fazer: "Sustenta 12 pessoas por 30 dias sem água externa", "Capacidade médica para trauma básico", "Geração independente de energia". Mais legível e motivador que score abstrato.
Compartilhamento de Inventário (Modelo Híbrido)
Privacidade por categoria
Itens de energy, communication e shelter são visíveis para o círculo por padrão — são recursos coletivos. Itens de food, water e medical são privados por padrão — expor escassez cria risco. Cada membro pode ajustar individualmente.
Prompt 4 — Circle Intelligence
Análise coletiva e sugestões de fortalecimento
Input: inventário agregado do círculo + cenários ativos dos membros. Output: gaps coletivos, distribuição de recursos, qual aquisição coletiva aumenta mais o score, e (no fluxo de aprovação) impacto do candidato no círculo.
Dependência: Fases 1–4 completas. Circle strength depende de inventário individual funcionando (Fase 3) e cross-scenario intelligence (Fase 4).
Entregáveis da Fase 5
Usuário pode criar um Círculo e convidar membros
Líder aprova/rejeita com score e impacto do candidato visíveis
Circle Strength Score calculado e atualizado em tempo real
Capability badges gerados com base no inventário coletivo
Modelo híbrido de privacidade por categoria funcionando
06
Resiliência
Offline Mode & PWA
Um sistema de emergência que fica offline durante emergências não é um sistema de emergência. Essa fase garante que tudo que importa funciona sem internet.
2 semanas
Service Worker & Cache Strategy
Cache das rotas críticas
Cache-first para: /dashboard, /checklist, /plan, /scenario. Network-first para: /circles (dados sociais podem ser stale). Stale-while-revalidate para: knowledge base chunks. Usar Workbox para gerenciar estratégias por rota.
IndexedDB para dados do usuário
Persistir localmente: último plano gerado, checklist completa com status de cada item, perfil + família. Sincronização com Supabase quando conexão retorna (conflict resolution: last-write-wins para inventário, merge para checklists).
Knowledge base chunks pré-carregados
Os top 50 chunks mais relevantes para os cenários ativos do usuário são baixados e armazenados no IndexedDB no momento que ele está online. Esses chunks alimentam o fallback offline do motor de decisão.
Indicadores de Estado do Sistema
Status indicator na nav (3 estados)
CONNECTED (verde) → DEGRADED (laranja, sem IA mas com cache) → OFFLINE (vermelho, somente local). Transição automática. O usuário sempre sabe em qual estado o sistema está operando.
PWA Configuration
Manifest + meta tags + ícones
Configurar manifest.json completo. Meta tags PWA conforme EOS Design System v1 (seção 11). Ícones para iOS e Android. Splash screens. Testar instalação como app em iOS Safari e Android Chrome.
Entregáveis da Fase 6
App instalável como PWA em iOS e Android
Checklists e planos acessíveis sem internet
Marcar itens como adquirido funciona offline e sincroniza ao voltar
Indicador de estado (Connected/Degraded/Offline) funcionando
Knowledge base chunks dos cenários ativos pré-carregados
07
Hardware
LoRa Mesh & Comunicação Descentralizada
A feature mais diferenciada do EOS. Comunicação entre dispositivos sem nenhuma infraestrutura. Relevante quando até o celular para de funcionar.
3 semanas
Hardware
Seleção e prototipagem do hardware
Módulo: ESP32 + LoRa SX1276 (868MHz para Brasil/Europa, 915MHz para EUA). Protocolo mesh: Meshtastic (open source, battle-tested). Alcance: 1–10km em linha reta. Bateria: 18650 Li-Ion, autonomia mínima 72h.
Integração App ↔ Hardware
Bridge BLE entre app e dispositivo
O smartphone se conecta ao ESP32 via Bluetooth LE. O app envia mensagens curtas (planos de ação, status de checklist, localização) que o ESP32 retransmite via LoRa para outros dispositivos EOS na área.
Protocolo de mensagens EOS
Formato comprimido para LoRa (payload máximo ~250 bytes). Tipos de mensagem: STATUS (estou bem/preciso de ajuda), RESOURCE (tenho X, preciso de Y), PLAN (compartilhar plano de ação), LOCATION.
UI do estado OFFLINE no app
Quando em modo LoRa: mostrar dispositivos detectados na rede mesh, mensagens recebidas, e interface simplificada de envio. O app funciona como terminal do hardware.
⚠ Esta fase requer hardware físico para testes. Iniciar a prototipagem eletrônica em paralelo com a Fase 5 para não atrasar o cronograma.
Entregáveis da Fase 7
Protótipo de hardware ESP32 + LoRa funcional
App conecta ao dispositivo via BLE e envia/recebe mensagens
Rede mesh entre 2+ dispositivos funcionando
UI do modo OFFLINE reflete comunicação LoRa ativa

Lançamento
Beta Fechado & Lançamento Público
Onboarding das famílias fundadoras, ajustes finais com feedback real, e lançamento público com a landing page ativa.
2 semanas
Beta Fechado
Onboarding das famílias da waitlist
Começar com 10–20 famílias em regiões de risco real. Acompanhar de perto: onde travam no onboarding, quais cenários mais ativam, taxa de completude do tier Essencial na primeira semana.
Refinamento baseado em uso real
Os primeiros usuários vão revelar gaps que nenhum teste interno encontra. Reservar 1 semana inteira para iteração rápida com base no feedback dessas famílias.
Lançamento Público
Landing page ao vivo com waitlist aberta
A landing page já existe e está refinada. Conectar o formulário de email a uma lista real (Resend ou similar). Ativar analytics (Vercel Analytics — privacy-first).
Monitoramento de produção
Alertas para: erro na chamada LLM, latência > 10s na geração de checklist, falha no cálculo do circle_strength. Logs estruturados no Vercel.
Entregáveis da Fase 8
10–20 famílias fundadoras ativas e com feedback documentado
Landing page ao vivo com waitlist funcional
Monitoramento de produção ativo
EOS v1.0 público
Princípio de execução: Cada fase entrega algo que funciona de ponta a ponta. Não existe "vamos integrar tudo no final". A Fase 3 já é um produto utilizável — com perfil, cenário e checklist gerada pela LLM. As fases seguintes adicionam inteligência, camadas sociais e resiliência de infraestrutura sobre uma base que já funciona.

Velocidade vs. perfeição: Conforme o blueprint de engenharia original — execução e clareza valem mais que perfeição. O fallback determinístico garante que o produto nunca quebra completamente. O refinamento é contínuo.
